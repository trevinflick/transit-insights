#
# ⚠️  DANGER — `crontab <file>` REPLACES EVERY CRON JOB FOR THE USER. ⚠️
#
# Only use the destructive `crontab cron/crontab.txt` form on a server where
# you are certain no other cron jobs exist. Anywhere else, the install will
# silently wipe every unrelated entry on the user's crontab. Use the
# marker-based merge below instead.
#
# Safe install procedure (works whether or not other jobs exist, and is
# idempotent — repeated runs leave one CTA-INSIGHTS block in the live
# crontab, never duplicates):
#
#   1. Snapshot the live crontab:
#        crontab -l > /tmp/live.crontab
#   2. Strip any existing CTA-INSIGHTS block from the live crontab:
#        awk '/^# CTA-INSIGHTS-START/{skip=1} /^# CTA-INSIGHTS-END/{skip=0; next} !skip' \
#          /tmp/live.crontab > /tmp/live.stripped
#   3. Extract just the marker block (NOT the docs above) from this file:
#        awk '/^# CTA-INSIGHTS-START/,/^# CTA-INSIGHTS-END/' cron/crontab.txt \
#          > /tmp/cta-block.txt
#   4. Substitute the repo path placeholder (cron does NOT expand variables
#      in R= assignments — must be inlined):
#        sed -i "s|/path/to/cta-insights|$(pwd)|g" /tmp/cta-block.txt
#   5. Concat + install:
#        cat /tmp/live.stripped /tmp/cta-block.txt | crontab -
#   6. Verify line counts pre/post match for unrelated jobs, and that the
#      installed R= line is a real absolute path (not "/path/to/..."):
#        crontab -l | grep "^R="
#
# IMPORTANT: only the content between # CTA-INSIGHTS-START and
# # CTA-INSIGHTS-END below gets installed by the procedure above. The
# install header (R= path, comments) lives INSIDE the markers so a
# repeated merge replaces them in place rather than stacking.
#
# CRON DOES NOT EXPAND VARIABLES IN ASSIGNMENT LINES. The R= value below
# uses a literal path placeholder (`/path/to/cta-insights`) that step 4 of
# the install procedure MUST substitute via sed before piping to crontab,
# e.g. `sed 's|/path/to/cta-insights|/home/cailin/Development/cta-insights|g'`.
# Earlier we used `R=$REPO_ROOT/bin/cron-run.sh` with REPO_ROOT defined on
# the line above, which compiles fine but cron stores R as the literal
# string `$REPO_ROOT/bin/cron-run.sh` and every job silently fails.

# All jobs run via bin/cron-run.sh <log-name> <script> [args...] which
# handles cd, timestamping, and log redirection. Substitute the path

R=/path/to/cta-insights/bin/cron-run.sh

# --- Trains ---
# train-pulse on even minutes (0,2,4,...) and train-alerts on odd minutes
# (1,3,5,...) so they never share a wall minute. Threading invariant: when
# both run on the same minute they each see "no thread parent yet" and post
# separate top-level posts instead of replying to one another.
0-58/2  *  *  *  *   $R train-pulse      bin/train/pulse.js
4-59/15 *  *  *  *   $R train-bunching   bin/train/bunching.js
1-59/2  *  *  *  *   $R train-alerts     bin/train/alerts.js
7-59/15 *  *  *  *   $R train-gaps       bin/train/gaps.js
0       */2 *  *  *  $R train-speedmap   bin/train/speedmap.js
0       */3 *  *  *  $R train-snapshot   bin/train/snapshot.js
8       *  *  *  *   $R train-ghosts     bin/train/ghosts.js
# incident-roundup runs every 5 min on :04 offset, separate from pulse :02 /
# alerts :02 / gaps :07 so its detector-correlation read isn't racing the
# writers. Reads meta_signals for both train and bus near-misses and posts a
# single rollup when distinct-source signals on the same line/route within
# 30 min sum past the threshold.
4-59/5  *  *  *  *   $R incident-roundup bin/incident-roundup.js

# --- Buses ---
1-59/20 *  *  *  *   $R bus-bunching     bin/bus/bunching.js
# bus-alerts on even minutes, bus-pulse on odd minutes — same threading-
# safety reasoning as train-alerts vs train-pulse. Bot accounts differ from
# the train side, so cross-bot collisions (e.g. bus-pulse + train-alerts both
# on odd) are fine; the rule is only within a single bot's thread.
0-58/2  *  *  *  *   $R bus-alerts       bin/bus/alerts.js
1-59/2  *  *  *  *   $R bus-pulse        bin/bus/pulse.js
6-59/20 *  *  *  *   $R bus-gaps         bin/bus/gaps.js
0       */2 *  *  *  $R bus-speedmap     bin/bus/speedmap.js
7       *  *  *  *   $R bus-ghosts       bin/bus/ghosts.js
# observe-buses runs every minute. CTA refreshes every 30s, so 60s misses at
# most one refresh cycle. 124 routes × 13 chunks × 1440 ticks/day ≈ 18.7k/day,
# well under the 100k/day cap. bunching/gaps/pulse all read the cache
# (maxStaleMs=90s) so observe-buses is the only API call site for the
# all-routes workload.
*       *  *  *  *   $R observe-buses    scripts/observeBuses.js
# observe-trains polls Train Tracker every 30s. Cron's minimum granularity is
# 1 min, so the script itself runs two ticks 30s apart per firing. One batched
# call returns all 8 lines, so 2880 calls/day — well under the 100k/day cap.
# At 30s cadence the snapshot timelapse gets ~30 frames per train per 15-min
# window, vs ~7-8 at the prior 2-min cadence — visibly continuous motion.
*       *  *  *  *   $R observe-trains   scripts/observeTrains.js
